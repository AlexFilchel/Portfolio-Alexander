## Project Summary

- Personal portfolio SPA built with **React 18** + **Vite 5**.
- Styling uses **Tailwind CSS 3**, **PostCSS**, and **Autoprefixer**.
- The codebase is **JavaScript + JSX only**.
- The repo uses **ES modules** (`"type": "module"` in `package.json`).
- There is **no TypeScript**, **no test runner**, **no linter**, and **no formatter config** at the repo root.

## Repository Structure

- `src/App.jsx` — app composition and top-level window/taskbar wiring.
- `src/components/` — reusable UI primitives and shared visual components.
- `src/components/desktop/` — desktop surface and icon layout.
- `src/components/taskbar/` — taskbar and search UI.
- `src/components/windows/` — window shell, resize, drag, and shared window chrome.
- `src/hooks/` — reusable stateful behavior, especially window management.
- `src/modules/apps/` — app-level screens rendered inside windows.
- `src/data/` — data-driven app registry and content.
- `src/utils/` — small helper utilities such as class name composition.
- `src/index.css` — base styles and global CSS.
- `tailwind.config.js` — Tailwind theme extensions and design tokens.
- `vite.config.js` — Vite config.

## Code Style: General

These are observed repository conventions and should be preserved.

- Use **2-space indentation**.
- Use **single quotes**.
- Use **semicolons**.
- Keep **trailing commas** in multiline arrays, objects, and argument lists.
- Prefer small, readable helper functions over clever inline logic.
- Keep modules focused; avoid giant mixed-responsibility files unless extending an existing pattern.

## Imports

- Keep imports at the top of the file.
- Group React imports first, then local imports.
- Use relative imports consistent with nearby files.
- Prefer explicit named imports where the module already exposes them.
- Do not introduce path aliases unless explicitly requested.
- Preserve existing import ordering style in touched files instead of reformatting unrelated imports.

## React Conventions

- Use **functional components** only.
- Prefer **named exports** for components, hooks, and utilities.
- Keep the existing **default export** pattern where it already exists (for example `App.jsx`).
- Extract reusable logic into hooks when behavior becomes stateful or interaction-heavy.
- Use `useMemo`, `useCallback`, and `useEffect` only when they provide clear value.
- Prefer deriving UI from state/data rather than duplicating state.

## State and Logic

- Central window behavior lives in `src/hooks/useWindowManager.js`; extend that flow instead of duplicating window state elsewhere.
- Keep data-driven app definitions in `src/data/apps.js`.
- Prefer updating configuration/data files for new content before adding hardcoded UI branches.
- Reuse existing helpers such as `classNames` before adding new utility patterns.

## Styling

- Prefer **Tailwind utility classes in JSX**.
- Reuse existing visual language: glassmorphism, Windows-11-inspired spacing, rounded corners, soft shadows.
- Extend `tailwind.config.js` only when the same design token or animation is reused.
- Use `src/index.css` for global/base styling only; avoid shifting component styling there unless it is truly shared.
- Do not introduce a new styling library.

## Naming

- Use **PascalCase** for React components and app modules.
- Use **camelCase** for variables, functions, hooks, and props.
- Hooks must start with `use`.
- Use descriptive handler names like `handleTaskbarClick`, `onOpenApp`, `updateWindowBounds`.
- Use uppercase constants for stable layout/config values when appropriate.

## Error Handling and Safety

- Follow the repo’s defensive style: guard early and return existing state when inputs are invalid.
- Avoid throwing new errors in UI flows unless there is a real exceptional condition.
- Preserve resilient checks such as optional chaining and null guards.
- When updating reducers or event handlers, keep state transitions explicit and predictable.
- Do not silently swallow meaningful failures if a user-facing fallback or guard is possible.

## Internationalization / Content

- The portfolio content is Spanish-oriented; preserve existing Spanish copy unless asked to change it.
- Keep locale-sensitive formatting consistent with existing `es-AR` usage.
- Maintain the current tone and UX labels unless the task explicitly changes copy.
